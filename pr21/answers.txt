1. Какие основные принципы REST архитектуры вы знаете?
- Ресурсо-ориентированность - все сущности представляются как ресурсы с уникальными URI
- Единообразие интерфейса - использование стандартных HTTP-методов (GET, POST, PUT, DELETE)
- Бессостояние - каждый запрос содержит всю необходимую информацию для его обработки
- Кэшируемость - ответы могут быть закэшированы для улучшения производительности
- Слоистая система - клиент не знает, подключен ли он к конечному серверу или к промежуточному
- Код на стороне клиента (опционально) - сервер может предоставлять исполняемый код клиенту

2. В чем разница между HTTP методами PUT и PATCH?
- PUT - полное обновление ресурса. Клиент отправляет полное представление ресурса, заменяя весь существующий ресурс
- PATCH - частичное обновление ресурса. Клиент отправляет только изменяемые поля, обновляя часть ресурса
- PUT требует отправки всех обязательных полей, PATCH - только тех полей, которые нужно изменить

3. Какие коды состояния HTTP наиболее часто используются в REST API?
- 200 OK - успешный запрос
- 201 Created - ресурс успешно создан
- 400 Bad Request - ошибка валидации или неверный запрос
- 404 Not Found - ресурс не найден
- 500 Internal Server Error - внутренняя ошибка сервера
- 204 No Content - успешный запрос без тела ответа
- 401 Unauthorized - требуется аутентификация
- 403 Forbidden - доступ запрещен

4. Что такое ресурсо-ориентированный дизайн URL?
Это подход к проектированию URL, где каждый URL представляет собой конкретный ресурс:
- /api/books - коллекция книг
- /api/books/1 - конкретная книга с ID=1
- Используются существительные вместо глаголов
- URL должны быть иерархическими и понятными
- Не используются глаголы в URL, так как действие определяется HTTP-методом

5. Какие преимущества предоставляет пагинация в API?
- Уменьшение нагрузки на сервер и базу данных
- Ускорение времени ответа API
- Экономия сетевого трафика
- Улучшение пользовательского опыта при больших объемах данных
- Предсказуемое потребление памяти на клиенте
- Возможность постраничной навигации

6. Как организовать валидацию данных в REST API?
- Валидация на стороне сервера (обязательная)
- Проверка обязательных полей
- Проверка формата данных (email, ISBN, даты и т.д.)
- Проверка типов данных и диапазонов значений
- Проверка уникальности данных
- Валидация бизнес-правил
- Возврат детальных сообщений об ошибках в едином формате

7. Какие подходы к обработке ошибок в REST API вы знаете?
- Использование соответствующих HTTP-статусов
- Единый формат ответов об ошибках
- Детальные сообщения для разработчиков
- Логирование ошибок на сервере
- Классификация ошибок (клиентские, серверные, бизнес-логики)
- Graceful degradation - корректная обработка частичных сбоев
- Ведение журнала аудита для отслеживания проблем

8. В чем разница между query parameters и path parameters?
- Path parameters - часть URL пути: /api/books/1 (обязательные, идентифицируют ресурс)
- Query parameters - параметры после знака ?: /api/books?page=1&limit=10 (опциональные)
- Path parameters идентифицируют конкретный ресурс или коллекцию
- Query parameters используются для фильтрации, сортировки, пагинации, поиска
- Path parameters иерархические, query parameters - плоские
- Path parameters обязательны для маршрута, query parameters - нет
